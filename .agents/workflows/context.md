# 🌐 JUK's Personal Portfolio - AI Context & Guidelines

본 문서는 JUK(사용자)의 포트폴리오 웹사이트를 개발하기 위해 AI 에이전트가 **필수적으로 숙지하고 따라야 할** 배경 지식, 핵심 용어, 참조 사이트 및 코딩 철학을 밀도 있게 정리한 지침서입니다.

---

## 1. 핵심 용어 정의 (Terminology)
본 프로젝트에서 AI와 사용자는 코스트 및 혼선을 줄이기 위해 아래와 같은 용어를 명확히 구분하여 사용합니다.

- **창 컴포넌트 (Window Component)**: 바탕화면 내에서 띄워지는 모바일/데스크탑 대응 패널. 브라우저 팝업이 아닌, DOM 상에 띄워지는 `div` 기반의 드래그/리사이즈 가능한 객체입니다.
- **1차 창 컴포넌트 (Primary Windows)**: 화면 하단의 내비게이션 바(Nav) 버튼들(ABOUT, WORK, CAREERS, CONTACT)을 눌렀을 때 나타나는 메인 창들입니다. 이미 맨 앞에 열려 있는 상태에서 Nav 버튼을 다시 누르면 데이터와 상태가 보존된 채 **최소화**(`visibility: hidden`) 처리되며, 창 상단의 X 버튼이나 제목 더블클릭으로 끄면 DOM과 위치/크기 상태가 완전히 **파괴(종료)**됩니다.
- **2차 창 컴포넌트 (Secondary Windows)**: 1차 창 컴포넌트 '내부 콘텐츠'를 클릭했을 때 나타나는 서브 창입니다. 1차 창이 닫혀도 2차 창은 별개의 객체(독립적 생명주기)로서 바탕화면에 계속 존재해야 합니다.
- **Singleton Pattern 규칙**: 1차든 2차든 똑같은 내용을 가진 창은 화면에 두 개 이상 중복으로 열릴 수 없습니다 (고유 ID 관리).

## 3. 기술 스택 원칙 및 최적화 철학 (Optimization First)
- **자유로운 프레임워크 선택권 (현재: Vanilla JS)**: 프레임워크 사용에 대한 제약은 없으나, 본 프로젝트의 핵심 로직(수많은 창의 드래그 및 1픽셀 단위 리사이징)은 화면을 매우 높은 빈도로 다시 그리는 작업입니다. Virtual DOM을 거치지 않고 DOM을 직접 컨트롤하는 순수 바닐라 환경이 훨씬 높은 FPS(초당 프레임) 확보에 유리하므로 현재 Vanilla(HTML/CSS/JS) 구조를 채택했습니다.
- **극단적인 최적화 방침**: 저사양 컴퓨터에서도 렉 없이 동작해야 하므로, 움직이는 컴포넌트는 `translate3d`를 통한 하드웨어 가속, `will-change: transform` 명시, 이벤트를 묶어 처리하는 `requestAnimationFrame` 적용이 강제됩니다.

## 4. UI/UX 세부 디테일 수칙
- **윈도우 사이즈 및 아이콘**: 초기 크기는 뷰포트에 비례하여 동적으로 결정 (1차 창: `width = vw*0.35`, `height = vh*0.75`, min/max clamp 적용). 블랙 타이틀바 및 오른쪽 끝에 1:1 비율의 깔끔한 인라인 SVG로 제작된 최대화(□), 닫기(✕) 아이콘 버튼이 위치합니다.
- **창 닫기 및 동작 분리**: 좌측 상단 Window 타이틀 텍스트를 더블클릭하거나 우상단 X 버튼을 누르면 창은 **완전히 종료(파괴)**되어 저장 중이던 크기/위치 상태가 사라집니다. 그러나 하단 Nav 버튼을 통해 끌 경우(최소화)에는 DOM 자체와 크기/위치, 심지어 최대화 상태(`isMaximized`)까지 고스란히 메모리에 **보존**된 채 화면에서만 숨겨집니다.
- **최대화/복원 (Maximize/Restore)**: 타이틀바의 빈 공간 더블클릭 또는 □ 버튼으로 최대화 토글. 최대화 상태에서는 JS 픽셀이 아닌 CSS `100vw`/`100vh`로 크기를 전환하여 브라우저 리사이징에도 완벽 대응. 최대화 상태에서 타이틀바를 1px이라도 드래그하면 즉시 최대화가 해제되며 마우스 커서 기준으로 복원(Breakout). 최대화 상태에서 가장자리 리사이즈 핸들을 잡으면 화면 크기만한 일반 상태로 강제 복귀 후 리사이즈 연동.
- **Nav 버튼 다이내믹 피드백**: 창 컴포넌트가 바탕화면에 열려(또는 복원되어) 있으면 대응하는 하단 Nav 버튼은 반투명해지며(`is-active`), 그 창이 최상위(Z-index 제일 앞)에 있을 때 Nav 버튼을 호버(Hover)하면, 버튼 텍스트가 즉각 `✕` 기호로 변경되며 버튼을 눌렀을 때 최소화될 것임을 확실히 직관적으로 알려줍니다. 이때 버튼 크기 변화(`Layout Shift`)를 막기 위해 CSS `color: transparent`와 `::after` 가상 요소를 씁니다.
- **창 스폰(Spawn) 위치 계산**: 1차 창 컴포넌트가 열릴 때는 하단 메뉴 버튼의 가로 위치(Index)에 비례하여 화면 좌우에 나란히 배치되고, 순서에 따라 대각선으로 자연스럽게 내려가도록 Y축 오프셋을 줍니다. 
- **상태 기억(Memory)**: 창이 (Nav가 아닌 다른 이유로 최소화되거나) 닫혔더라도 위치/크기가 유지되는 경우, 다시 열면 유저가 리사이즈하거나 옮겨놓았던 크기와 좌표값 그대로 열려야 합니다. 단, 크기가 310x210 이하로 너무 작아졌거나 화면 밖(뷰포트 바깥)에 꽂혀 구출 불가능해진 상태라면 이를 감지하고 기본 Spawning 설정으로 초기화하여 열어주어야 합니다.
- **모바일 대응 (<767px)**: 데스크탑의 '드래그 창' 패러다임이 버려집니다. 대신 기존 데스크탑에서 열어뒀던 여러 창들 중 제일 앞(가장 높은 Z-index)에 켜져 있던 창 객체 하나가 모바일 뷰포트를 꽉 채우는 패널/모달 형태로 변환되어 우선적으로 노출되어야 합니다.
- **유연한 빈 화면 드래그 허용**: 창 이동은 상단 타이틀바뿐만 아니라 내부 흰 본문의 빈 여백 공간을 잡고 드래그해도 지원됩니다. 단, 그 영역 안의 버튼이나 링크 등 인터랙티브 요소를 클릭하거나 텍스트를 복사하는 동작에는 절대 간섭해서는 안 됩니다.

## 4.5. 시스템 아키텍처 및 렌더링 세부 수칙 (New Architecture)
- **콘텐츠 아키텍처 (HTML Fragments)**: 창 내부 콘텐츠는 js 내부에 하드코딩되지 않고 `content/*.html` 파일로 분리되어 있습니다. 창 생성 시 `ContentLoader.js`가 `fetch`를 통해 비동기로 로드하며 로딩 스켈레톤을 지원합니다.
- **창 독립적 반응형 (Container Queries)**: 뷰포트에 의존하는 낡은 반응형(`@media`) 방식을 타파하고, 창 내부 `.window-content-area`에 `@container window-panel`을 선언했습니다. 이를 기반으로 설계하여 각 창의 너비 변화에 따라 내부 레이아웃(Masonry 갤러리의 1열~6열 확장 등)이 살아 숨쉬듯 독립적으로 변형됩니다.
- **Z-Index Layer 분리**: 1차 창 컴포넌트와 2차 창 컴포넌트는 별도의 Z-Index 체계를 가집니다. 1차 창은 100부터, 2차 창은 10000부터 시작하여 완전히 격리되며, 무조건 1차 창이 2차 창 뒤에 렌더링되어 UI 침범을 원천 차단합니다.
- **상태 동기화 이벤트 파이프라인**: 2차 창이 독립적으로 강제 종료(X 버튼 등)될 때 `windowClosed` 커스텀 이벤트를 Document에 발생시킵니다. 이를 통해 해당 창을 열리게 했던 주체(예: 뒤에서 대기 중인 갤러리 썸네일)가 즉시 이벤트를 리스닝하여 스스로의 `.is-active`(Dim) 상태를 해제하는 느슨한 결합 구조를 준수합니다.
